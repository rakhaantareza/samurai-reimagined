## 1. Executive Summary

**History of Samurai — Reimagined** is a complete redesign and rebuild of a basic web project originally created in 2022 as the final submission for Dicoding’s Basic Web Programming course.

The original website successfully fulfilled the submission requirements by using semantic HTML elements such as `header`, `footer`, `main`, `article`, and `aside`, displaying personal identity information, and implementing a layout with Flexbox or Float.

The new version will transform that simple informational website into an immersive, cinematic, and interactive historical experience about the samurai. The project will combine editorial web design, structured historical storytelling, responsive front-end development, and scroll-based animation.

The website should feel closer to a digital museum, interactive editorial feature, or historical exhibition than a traditional blog.

This project is intended to demonstrate visible growth from a beginner-level 2022 project into a polished 2026 front-end portfolio project.

---

## 2. Project Background

In 2022, the owner completed Dicoding’s Basic Web Programming course and created a website about the history of the samurai as the final submission.

The original submission was built primarily to satisfy the following requirements:

- Use semantic HTML elements including `header`, `footer`, `main`, `article`, and `aside`.
- Place appropriate content inside each semantic element.
- Display personal identity information inside an `aside` element.
- Use Flexbox or Float for page layout.
- Present an informational topic in a functional website.

The project was successfully completed, but its design and technical implementation reflect the owner’s skill level at the time.

The remake will preserve the original idea while completely rethinking the design, user experience, technical structure, content hierarchy, responsiveness, and interaction model.

## 3. Problem Statement

The original website is functional, but it has several limitations:

1. The visual hierarchy is basic and does not create a strong first impression.
2. The layout resembles a long-form article rather than an immersive experience.
3. The historical content is not arranged as a clear narrative journey.
4. The imagery lacks a consistent visual direction.
5. The website has limited interaction and no meaningful motion design.
6. The responsive experience is not yet strong enough for a modern portfolio.
7. The project does not fully represent the owner’s current front-end knowledge.
8. The website does not clearly communicate the growth between the 2022 version and the new version.
9. The current presentation is not strong enough to serve as a flagship portfolio project.
10. The project has not yet documented its design decisions, development process, challenges, accessibility considerations, or AI-assisted workflow.

## 9. Product Principles

The following principles should guide all design and development decisions.

### 9.1 Story first

Every section must contribute to the historical journey. Visual effects must support the story, not distract from it.

### 9.2 Strong even without animation

The static layout must already look polished before motion is added.

### 9.3 Editorial, not decorative

The project should avoid overusing obvious Japanese visual symbols. The design must feel curated rather than themed like a template.

### 9.4 Motion with purpose

Animation should improve focus, pacing, progression, or atmosphere.

### 9.5 Readability over spectacle

Text must remain readable. Contrast, line length, spacing, and responsive typography are more important than dramatic effects.

### 9.6 Mobile is a first-class experience

Mobile should not feel like a reduced desktop version. Layout and animation behavior should be intentionally adapted.

### 9.7 Historically responsible

Historical claims should be reviewed against credible sources. AI-generated imagery must not be presented as authentic historical photography.

### 9.8 Transparent AI usage

The project should clearly state that AI assisted with planning, drafting, visual generation, code implementation, debugging, and refinement.

## 26. Risks and Mitigation

### Risk 1 — Scope becomes too large

**Mitigation:**  
Do not add WebGL, 3D, audio, or multi-page architecture before Version 1.0 is complete.

### Risk 2 — AI-generated design feels generic

**Mitigation:**  
Use a locked visual direction, review every section, and refine outputs based on screenshots rather than accepting the first result.

### Risk 3 — Historical content is inaccurate

**Mitigation:**  
Use credible sources, verify claims, and clearly distinguish legend from history.

### Risk 4 — Animation hurts performance

**Mitigation:**  
Build the static layout first, animate only important elements, test mobile performance, and simplify motion on smaller devices.

### Risk 5 — Website looks good only on desktop

**Mitigation:**  
Create a dedicated mobile wireframe and test each section at mobile widths before proceeding.

### Risk 6 — The owner cannot explain AI-generated code

**Mitigation:**  
Review each major file, request explanations from Codex, document technical decisions, and work through small milestones.

### Risk 7 — The project remains unfinished

**Mitigation:**  
Use small commits, fixed milestones, clear acceptance criteria, and avoid adding optional features before launch.
